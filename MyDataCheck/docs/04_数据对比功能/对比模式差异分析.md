## 问题发现

### 问题1: 完整模式是否输出接口请求失败文件？

**答案：✅ 会输出**

完整模式在 `fetch_api_data` 阶段会输出接口请求失败的记录：

```python
# fetch_api_data.py 第806-810行
if output_error_records and errors:
    error_csv_path = output_csv_path.replace('.csv', '_error_records.csv')
    self._write_error_records_csv(error_csv_path, headers, rows, errors)
    print(f"✅ 请求失败记录文件写入完成: {error_csv_path}")
```

**输出文件名：** `{output_prefix}_api_data_error_records.csv`

**配置控制：**
```json
{
  "output_config": {
    "output_error_records": true  // 默认为true
  }
}
```

### 问题2: 接口请求失败的记录是否计入特征统计？

**答案：❌ 不应该计入，但完整模式有BUG！**

#### 流式模式（正确）✅

```python
# streaming_comparator.py 第330-370行
if "error" in comparison_result:
    # 错误行：跳过，不写入分析报告，只记录到错误列表
    error_rows += 1
    # ... 不更新 feature_stats
else:
    # 正常行：写入分析报告
    success_rows += 1
    # 更新特征统计（只统计成功的行）
    for header in feature_headers:
        feature_stats[header]["total"] += 1
        if header in comparison_results:
            feature_stats[header]["mismatch"] += 1
        else:
            feature_stats[header]["match"] += 1
```

**流式模式逻辑：**
- ✅ 接口失败的记录 → 不计入 `feature_stats`
- ✅ 只统计成功请求的记录

---

#### 完整模式（有问题）❌

```python
# compare_api_data_optimized.py 第280-310行
key = (cust_no, use_create_time)

# 在索引中查找
if key not in api_index:
    unmatched_count += 1
    continue  # ❌ 这里跳过了，但没有区分是"接口失败"还是"数据不匹配"

matched_count += 1
api_data = api_index[key]

# 对比特征
for j, header in enumerate(original_feature_headers):
    # 更新统计
    feature_stats[header]["total"] += 1  # ⚠️ 只有在索引中找到的才会计入
```

**完整模式问题：**

1. **接口失败的记录会被写入 `api_data.csv`**
   ```python
   # fetch_api_data.py 第865行
   if i in errors:
       # 错误行：只填充原始数据（仍然会写入文件）
       for j in range(min(len(rows[i]), len(output_headers))):
           row_data[j] = rows[i][j]
   ```

2. **但在对比阶段，接口失败的记录不会被加入索引**
   ```python
   # compare_api_data_optimized.py 第180-210行
   def _build_lightweight_index(...):
       for i, row in enumerate(rows):
           # 只有成功的记录才会被加入索引
           # 失败的记录在 api_data.csv 中，但不在索引中
   ```

3. **结果：接口失败的记录不会计入统计**
   - 原始CSV有100条记录
   - 接口失败10条
   - 对比时只能找到90条（索引中只有90条）
   - `feature_stats["total"]` = 90（不是100）

## 实际影响

### 场景示例

假设：
- 原始CSV：1000条记录
- 接口成功：950条
- 接口失败：50条

#### 完整模式输出

**文件1: `api_data.csv`**
- 包含1000行（950成功 + 50失败）
- 失败的行只有原始数据，没有接口返回值

**文件2: `api_data_error_records.csv`**
- 包含50条失败记录

**文件3: `feature_stats.csv`**
- 比对数据条数：950（只统计成功的）
- ✅ 正确：失败的50条不计入统计

#### 流式模式输出

**文件1: `analysis_report.csv`**
- 只包含有差异的记录（成功的950条中的差异）

**文件2: `errors.csv`**
- 包含50条失败记录

**文件3: `feature_stats.csv`**
- 比对数据条数：950（只统计成功的）
- ✅ 正确：失败的50条不计入统计

## 结论

### 问题1答案：✅ 完整模式会输出接口请求失败文件

- 文件名：`{output_prefix}_api_data_error_records.csv`
- 配置项：`output_config.output_error_records`（默认true）

### 问题2答案：✅ 接口请求失败的记录不会计入统计（两种模式都正确）

**完整模式逻辑：**
1. 接口失败的记录会写入 `api_data.csv`（只有原始数据）
2. 但不会加入索引
3. 对比时找不到索引，直接跳过（`unmatched_count += 1`）
4. 不计入 `feature_stats`

**流式模式逻辑：**
1. 接口失败的记录不写入 `analysis_report.csv`
2. 只记录到 `errors.csv`
3. 明确标记为 `error_rows`
4. 不计入 `feature_stats`

**两种模式都是正确的，只是实现方式不同：**
- 完整模式：通过"索引中找不到"来排除失败记录
- 流式模式：通过明确的错误标记来排除失败记录

## 建议

### 代码改进建议

完整模式的代码可以更清晰：

```python
# 建议在 compare_api_data_optimized.py 中添加注释
if key not in api_index:
    # 注意：这里包含两种情况：
    # 1. 接口请求失败（在api_data.csv中但不在索引中）

    continue
```

在文档中明确说明：
1. 接口失败的记录会输出到单独的错误文件
2. 接口失败的记录不会计入特征统计
3. `feature_stats` 中的"比对数据条数"只包含成功请求的记录
