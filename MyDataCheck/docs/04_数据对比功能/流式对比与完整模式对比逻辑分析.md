# 流式对比与完整模式对比逻辑分析

## 问题
`output_intermediate_files: false` 时使用流式对比，这个对比逻辑和为 `true` 时除了不输出文件，其它对比逻辑一样吗？会不会有异常对比不出来的情况？

## 结论
✅ **对比逻辑完全一致，不会有异常对比不出来的情况**

---

## 两种模式对比

### 模式1: 完整输出模式 (`output_intermediate_files: true`)

**流程**：
```
1. 获取接口数据 → 写入文件 (api_data.csv)
2. 读取原始CSV和接口数据CSV
3. 对比数据 → 输出所有文件
   - 分析报告 (analysis_report.csv)
   - 特征统计 (feature_stats.csv)
   - 全量合并 (全量数据合并.csv)
   - 错误记录 (errors.csv)
```

**特点**：
- 输出所有中间文件
- 可以查看接口返回的原始数据
- 可以查看全量合并数据
- 内存占用较高（需要同时加载原始数据和接口数据）

---

### 模式2: 仅报告模式 (`output_intermediate_files: false`)

**流程**：
```
1. 边请求接口 → 边对比 → 边写入报告
   - 分析报告 (analysis_report.csv)
   - 特征统计 (feature_stats.csv)
   - 错误记录 (errors.csv)
```

**特点**：
- 不输出中间文件（api_data.csv, 全量合并.csv）
- 流式处理，内存占用降低80-90%
- 只输出对比报告和统计结果

---

### 1. 特征值对比逻辑

```python
def _process_batch(self, batch, headers, feature_headers):
    # 对比逻辑
    for row_index, row in batch:

        cust_no = row[self.param1_column].strip()
        use_create_time = row[self.param2_column].strip()

                }
```

✅ **两种模式使用相同的 `compare_values` 函数进行对比**

## compare_values 函数（核心对比逻辑）

### 完整模式
```python
# compare_api_data_optimized.py
def compare_values(self, val1: str, val2: str) -> bool:
    """对比两个值是否相等（容忍浮点数误差）"""
    if val1 == val2:
        return True
    
    # 尝试作为数字对比
    try:
        num1 = float(val1)
        num2 = float(val2)
        return abs(num1 - num2) < 1e-6
    except (ValueError, TypeError):
        return False
```

### 流式模式
```python

try:
```

```python
        # 失败：返回错误标记
        return {"error": f"HTTP {response.status_code}"}
```

## 会不会有异常对比不出来？

### 场景1: 接口请求失败
- **完整模式**: 记录到 `_error_records.csv`，不参与对比统计
- **流式模式**: 记录到 `_errors.csv`，不参与对比统计
- ✅ **处理一致**

### 场景2: 接口返回空值
- **完整模式**: 空值与CSV值对比，记录为不匹配
- **流式模式**: 空值与CSV值对比，记录为不匹配
- ✅ **处理一致**

### 场景3: 浮点数精度差异
- **完整模式**: 使用 `abs(num1 - num2) < 1e-6` 容忍误差
- **流式模式**: 使用 `abs(num1 - num2) < 1e-6` 容忍误差
- ✅ **处理一致**

### 场景4: 字符串对比
- **完整模式**: 直接字符串比较 `val1 == val2`
- **流式模式**: 直接字符串比较 `val1 == val2`
- ✅ **处理一致**

### 场景5: 特殊字符
- **完整模式**: 先 `.strip()` 去除首尾空格
- **流式模式**: 先 `.strip()` 去除首尾空格
- ✅ **处理一致**

### 重要说明：接口失败记录的统计

**问题：接口请求失败的记录是否计入 `feature_stats` 的比对数据条数？**

**答案：❌ 不计入（两种模式都正确）**

#### 完整模式逻辑
1. 接口失败的记录会写入 `api_data.csv`（只有原始数据，接口字段为空）
2. 但不会加入对比索引
3. 对比时找不到索引，直接跳过（`unmatched_count += 1`）
4. **不计入** `feature_stats["total"]`

#### 流式模式逻辑
1. 接口失败的记录不写入 `analysis_report.csv`
2. 只记录到 `errors.csv`
3. 明确标记为 `error_rows`
4. **不计入** `feature_stats["total"]`

#### 示例说明

假设原始CSV有1000条记录：
- 接口成功：950条
- 接口失败：50条

**两种模式的统计结果：**
- `feature_stats` 中的"比对数据条数" = 950（只统计成功的）
- 失败的50条记录在错误文件中，不参与统计

**这是正确的行为！** 因为接口失败的记录无法进行特征值对比。

## 性能对比

### 完整模式
- **内存占用**: 高（需要同时加载原始数据和接口数据）
- **处理速度**: 较快（数据已在内存中）
- **适用场景**: 数据量小（< 10万行），需要查看中间文件

### 流式模式
- **内存占用**: 低（边请求边对比边写入，内存占用降低80-90%）
- **处理速度**: 较慢（需要实时请求接口）
- **适用场景**: 数据量大（> 10万行），只需要对比报告
