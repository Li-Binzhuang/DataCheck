# 只对比共同特征列错位修复说明

## 🐛 问题描述

**发现时间：** 2026-03-13
**问题：** 启用"只对比共同特征"后，对比的列出现错位

## 🔍 问题原因

### 原始错误逻辑

```python
# 1. 提取特征列（保留原始顺序和索引）
feature_cols_sql = headers_sql[sql_feature_start:]  # ['age', 'income', 'pt', 'score']
feature_cols_api = headers_api[api_feature_start:]  # ['age', 'income', 'score']

# 2. 筛选共同特征（错误：直接修改了列表）
feature_cols_sql = [col for col in feature_cols_sql if col in common_features]  
# 结果：['age', 'income', 'score']  ← 列表被修改了！

# 3. 构建特征映射（错误：使用筛选后的索引）
for idx, feature_sql in enumerate(feature_cols_sql):
    actual_sql_idx = sql_feature_start + idx  # ← 这里的idx是筛选后的位置！
    # idx=0 → 'age' → actual_sql_idx=1 ✓ 正确
    # idx=1 → 'income' → actual_sql_idx=2 ✓ 正确
    # idx=2 → 'score' → actual_sql_idx=3 ✗ 错误！应该是4
```

### 问题分析

**原始文件结构：**
```
file1: [id(0), age(1), income(2), pt(3), score(4)]
                ↑        ↑         ↑      ↑
         feature_start=1
```

**筛选后的列表：**
```
feature_cols_sql = ['age', 'income', 'score']  # pt被移除
索引位置：          [0,     1,        2]       # 新的索引位置
```

**错误的映射：**
```
'age'    → sql_feature_start + 0 = 1 ✓ 正确（原始位置也是1）
'income' → sql_feature_start + 1 = 2 ✓ 正确（原始位置也是2）
'score'  → sql_feature_start + 2 = 3 ✗ 错误！原始位置是4，不是3
```

**结果：** `score`列被错误地映射到了`pt`列的位置（索引3），导致对比错位！

## ✅ 修复方案

### 修复后的正确逻辑

```python
# 1. 提取特征列（保留原始顺序和索引）
feature_cols_sql = headers_sql[sql_feature_start:]  # ['age', 'income', 'pt', 'score']
feature_cols_api = headers_api[api_feature_start:]  # ['age', 'income', 'score']

# 2. 计算共同特征集合（不修改列表）
if compare_common_features_only:
    common_features = set(feature_cols_sql) & set(feature_cols_api)
    # common_features = {'age', 'income', 'score'}
else:
    common_features = None

# 3. 构建特征映射（在映射时才筛选）
for idx, feature_sql in enumerate(feature_cols_sql):
    # 如果启用筛选，跳过不在共同特征中的列
    if common_features is not None and feature_sql not in common_features:
        continue  # 跳过 'pt'
    
    actual_sql_idx = sql_feature_start + idx  # 使用原始索引！
    # idx=0 → 'age' → actual_sql_idx=1 ✓
    # idx=1 → 'income' → actual_sql_idx=2 ✓
    # idx=2 → 'pt' → 跳过
    # idx=3 → 'score' → actual_sql_idx=4 ✓ 正确！
```

### 关键改动

**修改前（错误）：**
```python
# 提前修改列表
feature_cols_sql = [col for col in feature_cols_sql if col in common_features]
feature_cols_api = [col for col in feature_cols_api if col in common_features]

# 使用修改后的索引（错误）
for idx, feature_sql in enumerate(feature_cols_sql):
    actual_sql_idx = sql_feature_start + idx  # 错误的索引
```

**修改后（正确）：**
```python
# 只保存共同特征集合，不修改列表
common_features = set(feature_cols_sql) & set(feature_cols_api) if compare_common_features_only else None

# 在构建映射时才筛选，保持原始索引
for idx, feature_sql in enumerate(feature_cols_sql):
    if common_features is not None and feature_sql not in common_features:
        continue  # 跳过非共同特征
    actual_sql_idx = sql_feature_start + idx  # 使用原始索引
```

## 🎯 核心原则

**关键点：** 在筛选共同特征时，必须保持原始列在文件中的索引位置，不能因为筛选而改变索引。

**正确做法：**
1. 提取特征列列表（保持原始顺序）
2. 计算共同特征集合（不修改列表）
3. 在构建映射时，使用原始索引，跳过非共同特征

**错误做法：**
1. 提取特征列列表
2. 直接修改列表，移除非共同特征
3. 使用修改后的索引（导致错位）

## 📝 总结

这是一个典型的"索引错位"问题。在处理列筛选时，必须区分"逻辑位置"和"物理位置"：
- **逻辑位置**：在筛选后的列表中的位置
- **物理位置**：在原始文件中的实际列索引

对比时必须使用物理位置，而不是逻辑位置。修复后的代码通过在映射构建阶段才进行筛选，确保使用的始终是原始文件中的物理索引。
