# 特征统计排序修复 - 快速参考

## 🎯 修复内容

接口数据对比模块输出的 `*_compare_feature_stats.csv` 文件中，有异常的特征现在按**异常率**（而不是异常数量）降序排序。

## 📝 修改的文件

1. `MyDataCheck/api_comparison/job/compare_api_data_optimized.py`
2. `MyDataCheck/common/report_generator.py`

## 🔄 排序变化

### 修改前（按异常数量排序）

```
特征A: 1000个异常 (异常率20%)  ← 排第1
特征B: 500个异常  (异常率50%)  ← 排第2
特征C: 80个异常   (异常率80%)  ← 排第3
```

### 修改后（按异常率排序）

```
特征C: 80个异常   (异常率80%)  ← 排第1 ✅
特征B: 500个异常  (异常率50%)  ← 排第2 ✅
特征A: 1000个异常 (异常率20%)  ← 排第3 ✅
```

## 💡 为什么这样更好？

- ✅ **更直观**: 一眼看出哪些特征质量问题最严重
- ✅ **更公平**: 不会因为数据量大就排在前面
- ✅ **更实用**: 优先关注异常比例高的特征

## 📊 输出文件示例

```csv
特征名,是否有异常,比对数据条数,匹配数量,异常数量,匹配率(%),异常率(%)
特征统计,,,,,,
无异常特征总数,50,,,,
有异常特征总数,30,,,,
,,,,,

# 无异常特征
feature_ok_1,无异常,1000,1000,0,100.00,0.00
feature_ok_2,无异常,1000,1000,0,100.00,0.00

# 有异常特征（按异常率降序）
feature_bad_1,有异常,100,10,90,10.00,90.00    ← 异常率最高
feature_bad_2,有异常,1000,500,500,50.00,50.00
feature_bad_3,有异常,10000,9000,1000,90.00,10.00
```

## ✅ 验证方法

1. 运行接口对比任务
2. 打开生成的 `*_compare_feature_stats.csv` 文件
3. 查看有异常特征部分的最后一列（异常率）
4. 确认异常率从高到低排列
