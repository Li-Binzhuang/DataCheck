# 数据对比功能优化说明 v1.3

## 📋 更新概述

本次更新优化了输出文件中主键值的获取逻辑，优先从 Sql 文件（文件1）获取主键值，如果没有则从接口文件（文件2）获取。

---

## ✨ 更新内容

### 主键值获取逻辑优化

**问题描述**：
- 原逻辑只从离线文件（接口文件/文件2）获取主键值
- 当主键值在在线文件（Sql文件/文件1）中更完整或更准确时，无法使用

**优化方案**：
- 优先从在线文件（Sql文件/文件1）获取主键值
- 如果在线文件中主键值为空，则从离线文件（接口文件/文件2）获取
- 确保输出文件中的主键值始终有值

**影响范围**：
- 差异数据明细文件
- 差异特征汇总文件
- 特征统计文件
- 全量数据合并文件

## 🔧 技术实现

### 代码位置
`MyDataCheck/online_comparison/job/data_comparator.py`

### 修改前逻辑
```python
# 直接使用离线文件的主键值
key_value = str(row_offline[offline_key_idx]).strip() if row_offline[offline_key_idx] is not None else ""

# 使用这个 key_value 记录差异
differences_dict[(key_value, feature_name)] = (offline_value, online_value, cust_no, ...)
```

### 修改后逻辑
```python
# 先获取离线文件的主键值（用于匹配）
key_value_offline = str(row_offline[offline_key_idx]).strip() if row_offline[offline_key_idx] is not None else ""

# 查找在线文件中对应的记录
online_row = None
for row_online in rows_online:
    if online_key_idx < len(row_online) and str(row_online[online_key_idx]).strip() == key_value_offline:
        online_row = row_online
        break

# 获取主键值：优先从在线文件（Sql文件）获取
key_value = ""
if online_row and online_key_idx < len(online_row) and online_row[online_key_idx] is not None:
    key_value = str(online_row[online_key_idx]).strip()

# 如果在线文件中没有主键值，使用离线文件的主键值
if not key_value:
    key_value = key_value_offline

# 使用优化后的 key_value 记录差异
differences_dict[(key_value, feature_name)] = (offline_value, online_value, cust_no, ...)
```

## 📊 使用场景

### 场景1：主键值在两个文件中都存在

**文件结构**：
```
Sql文件（file1）:
| apply_id | feature1 | feature2 |
|----------|----------|----------|
| A001     | 100      | 200      |

接口文件（file2）:
| apply_id | feature1 | feature2 |
|----------|----------|----------|
| A001     | 100      | 201      |
```

**结果**：
- ✅ 主键值使用 Sql 文件的值：A001
- ✅ 差异明细中显示：主键值=A001, feature2差异（200 vs 201）

### 场景2：主键值只在接口文件中存在

**文件结构**：
```
Sql文件（file1）:
| id | feature1 | feature2 |
|----|----------|----------|
|    | 100      | 200      |

接口文件（file2）:
| apply_id | feature1 | feature2 |
|----------|----------|----------|
| A001     | 100      | 201      |
```

**说明**：
- Sql 文件的主键列配置为 0（id列），但该列为空
- 接口文件的主键列配置为 0（apply_id列），有值

**结果**：
- ✅ 主键值使用接口文件的值：A001
- ✅ 差异明细中显示：主键值=A001, feature2差异（200 vs 201）

### 场景3：主键值格式不同

**文件结构**：
```
Sql文件（file1）:
| apply_id      | feature1 |
|---------------|----------|
| A001-20250126 | 100      |

接口文件（file2）:
| apply_id | feature1 |
|----------|----------|
| A001     | 101      |
```

**说明**：
- 两个文件的主键值格式不同
- 通过某种方式匹配成功（例如截取前4位）

**结果**：
- ✅ 主键值使用 Sql 文件的完整值：A001-20250126
- ✅ 差异明细中显示更详细的主键信息

## 💡 优势说明

### 1. 数据完整性
- ✅ Sql 文件通常包含更完整的主键信息
- ✅ 避免因接口返回的主键值被截断或简化而丢失信息

### 2. 数据追溯性
- ✅ 使用 Sql 文件的主键值，更容易在数据库中追溯原始记录
- ✅ 便于后续的数据分析和问题排查

### 3. 一致性
- ✅ 与 cust_no 的获取逻辑保持一致（也是优先从 Sql 文件获取）
- ✅ 统一的数据来源优先级规则

### 4. 容错性
- ✅ 如果 Sql 文件的主键值为空，自动使用接口文件的值
- ✅ 确保输出文件中的主键值始终有值

## 📝 注意事项

1. **主键匹配**：
   - 两个文件的主键值必须能够匹配（相等）
   - 匹配时使用的是离线文件（接口文件）的主键值
   - 匹配成功后，输出时使用在线文件（Sql文件）的主键值

2. **主键列配置**：
   - 确保正确配置两个文件的主键列索引
   - 主键列索引从 0 开始（A列=0，B列=1，...）

3. **数据完整性**：
   - 如果 Sql 文件的主键值为空，会自动使用接口文件的值
   - 确保至少有一个文件的主键值不为空
